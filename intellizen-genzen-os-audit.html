01 · Verdict

Yes — the chassis is already generic. The entity layer isn't.

IntelliZen can become the GenZen-wide business OS, and it is much closer than its own documentation admits: the CLAUDE.md still describes a 7-screen intel tool shelling out to claude -p, while the actual build has 14 routes, Notion-style generic databases on workspace.records (with revisions, an append RPC, and an audit log), a 13-workflow registry with approval gates, a 50-tool MCP surface, and a Hermes/Fiona agent execution path with a durable fallback queue. The hard parts of a business OS — flexible records, agent write-protocol, workflow receipts — exist.

The single biggest thing standing in the way: entity is a cosmetic label, not a structural dimension. The taxonomy defines four entities, but it lives only as JSON metadata on 9 databases; the intel layer has no entity column at all, the workflow registry stores free-text entity names ("GenZen HQ", "IntelliZen"), the one "operating view" is a hardcoded GenZen-Solutions keyword filter, and GoKart Studio has zero records anywhere in the system. Until entity becomes a first-class, queryable dimension that scopes every surface, adding a second entity means either polluting GZS views or forking the app. Everything else — the dual data layer, the missing internal search, the CRM-as-table — is real but downstream of that.

Architecture & data model 3/5 The workspace.records engine is genuinely load-bearing; the dual data layer and client-side invariants will fracture under scale. evidence ▾
  • Load-bearing and good: 9 namespaced schemas; generic databases/views/records with per-record revisions (20260702102017), deleted-record restore, an atomic section-append RPC (20260702090246) that already solved the concurrent-agent clobber problem, and an append-only workspace.work_events log.
  • Fracture line #1 — two implementations of one contract: src/lib/data.ts (5,198 lines) and mcp-server/src/index.ts (3,381 lines) independently reimplement records, agent work, workflows, and receipts. Every new object type must be built twice and will drift.
  • Fracture line #2 — invariants live in the client: relation reciprocity (data.ts:4878), revision snapshots, and the anchors↔workspace dual-write sync (syncOperationalWorkspaceDatabases, data.ts:2527) are TypeScript-side. Any writer that isn't the app — raw SQL, a new agent — silently skips them.
  • Fracture line #3 — legacy bridge: Projects/Operations exist twice (anchors.projects, 15 rows, and a mirrored workspace database) held together by a timed client-side sync.
  • Layout state is device-local: Home dashboard pins live in localStorage (src/lib/home-pins.ts:16-36) — invisible to agents and lost across machines.
UX & operator velocity 3/5 Strong primitives (pins, palette, dictation, GenUI) — but the operator still can't search their own company from inside the app. evidence ▾
  • Genuinely fast: pinnable Home dashboard, command palette, agent chat with streaming + generative-UI widgets, live dictation into the composer, peek panels.
  • The big manual tax — no internal search. The Search view is Exa web search only (src/views/Search.tsx:10); the command palette only navigates and triggers actions (command-palette.tsx:85-103). 590 knowledge documents, 19,110 chunks, 246 workspace records, and 1,303 signals cannot be found from inside the app.
  • Manual handoffs: investigation → report → client record → distribution asset are separate surfaces with no carried context; linking a case to a Clients record is by hand.
  • CRM upkeep is raw record editing — no pipeline board, no next-action nudges for the 4 introducers the distribution strategy depends on.
  • Docs drift costs agent velocity too: CLAUDE.md describes 7 screens and a Tauri-shell Claude integration that no longer exists — every agent session starts with a wrong map.
Agent-readiness 3/5 Best-in-class work protocol (claim/receipt/approve) — but agents can't touch CRM or any new database without dropping to raw SQL. evidence ▾
  • The protocol is real: ~50 MCP tools including claim_agent_work, append_agent_work_note, close_agent_work, request/resolve_workflow_approval, delegate_work (mcp-server/src/index.ts:1692-2504) — claim → receipt → approval is a working loop.
  • Resilient dispatch: app-initiated agent work tries the Hermes run queue, falls back to webhook, then to the durable comms.fiona_inbox (35 rows) — with the dispatch error surfaced (src/services/agent.ts:340-365).
  • The hole: no generic record tools. The MCP server exposes Tasks/Biz Ops/Workflows by hardcoded IDs but has no list_databases / query_records / create_record. Agents managing CRM, Clients, or Introducers must use raw SQL through the supabase-db MCP — bypassing revisions, reciprocity, and work_events. The intellizen-supabase-workspace skill exists precisely to compensate for this gap.
  • Audit trail unused: workspace.work_events has 2 rows — receipts are being written into record bodies, not the event log.
  • Known MCP footgun: list_project_signals returns full raw_payload — documented to exceed 1M chars on seeded projects.
Reliability & data safety 4/5 The 2026-07-03 audit hardening is real: anon-key client, scoped RLS, build guards, revisions, atomic appends. Residual risk is drift, not disaster. evidence ▾
  • Credential posture fixed: client runs on the anon key with server-side scoping to 22 app tables (src/lib/supabase.ts:3-8); the vite guard refuses production builds containing the service key; scripts/check-bundle-secrets.sh is the authoritative pre-publish gate.
  • Write safety: atomic append_record_section RPC prevents concurrent-agent clobbering; record revisions + trash restore; RLS enabled on every table checked.
  • Partial-data resilience is a stated and observed design rule (null publish dates, sparse Deep Research output, raw_payload preservation).
  • Residual risks: the anchors↔workspace dual-write sync can race (client-timed, 10-min window, data.ts:177-199); Home layout in localStorage is unbacked-up state; silent Hermes degradation to queue is logged but easy to miss; client-side invariants mean raw-SQL writers can corrupt relational integrity without any guard firing.

06 · The five highest-leverage moves

Ranked, each with a first step you can take this week

1Make entity a real dimension, not a JSON label
Effort: Mschema + shellunlocks every other move

Problem: entity exists only as taxonomy metadata on 9 workspace databases (taxonomy.ts:3-8). The intel layer has no entity column, the Workflow Registry holds free-text names ("GenZen HQ", "IntelliZen"), the operating view is a hardcoded GZS keyword filter (operating-views.ts:89-106), and nothing in the shell scopes by entity.

Change: a canonical workspace.entities table; an entity FK on anchors.projects, intel.monitors, intel.investigations, and workspace.databases; normalize the 13 registry rows to slugs; then a global entity filter in the app shell that every list query respects.

Cost of not doing: the moment GoKart data arrives, it pollutes every GZS view; agents keep guessing entity from strings; the "operating view" pattern can never be replicated for a second entity.

First step — this week

One additive migration + one data fix (~1 hour):

create table workspace.entities (
  slug text primary key, label text not null, status text not null default 'active');
insert into workspace.entities values
  ('genzen','GenZen','active'), ('genzen_solutions','GenZen Solutions','active'),
  ('gokart_studio','GoKart Studio','active'), ('founder_context','Founder Context','active');
alter table anchors.projects       add column entity text references workspace.entities;
alter table intel.monitors         add column entity text references workspace.entities;
alter table intel.investigations   add column entity text references workspace.entities;
-- backfill existing rows to 'genzen_solutions'; normalize the 13
-- Workflow Registry records' workflow_entity values to slugs
2Give agents generic record tools — stop the raw-SQL bypass
Effort: S–Mmcp-serverunlocks CRM automation + GoKart onboarding

Problem: the MCP server drives Tasks/Biz Ops/Workflows via hardcoded database IDs but has no list_databases, query_records, create_record, or update_record. Agents working CRM, Clients, Introducers — or any future database — drop to raw SQL through the supabase-db MCP, bypassing revisions, relation reciprocity, and work_events. A compensating skill (intellizen-supabase-workspace) exists solely because of this gap.

Change: five generic tools in mcp-server/src/index.ts that route through the same invariant logic the app uses, with entity + database scoping from move 1.

Cost of not doing: every new database silently forks the write path; relational integrity decays invisibly; CRM stays human-only.

First step — this week

Add the two read-only tools first — list_databases (id, name, entity, schema fields) and query_records (filtered, slim fields, no raw payloads) — to mcp-server/src/index.ts, rebuild, and let agents navigate the full catalog safely before any write tool lands.

3Collapse the dual data layer; push invariants into Postgres
Effort: Ldata.ts + mcp-server + RPCsunlocks build-once objects

Problem: src/lib/data.ts (5,198 lines) and mcp-server/src/index.ts (3,381 lines) independently implement the same contracts, and the invariants that keep records coherent — relation reciprocity (data.ts:4878), revision snapshots, the anchors↔workspace dual-write (data.ts:2527) — run only in whichever client remembered to implement them.

Change: move invariants into Postgres RPCs/triggers (the append_record_section RPC from migration 20260702090246 is the proven template), then extract a shared TypeScript contracts package consumed by both the app and the MCP server.

Cost of not doing: every capability costs 2× forever, and app-vs-agent drift becomes a data-integrity bug class you can't lint for.

First step — this week

Port updateWorkspaceRelationLinks to a workspace.update_relation_links RPC (mirroring the append RPC's pattern), point data.ts:4878 at it, and make it the only way relations change — one migration, one function swap.

4Internal search: one query across records, knowledge, and signals
Effort: MSQL function + Search tab + paletteunlocks "the app is where you find company state"

Problem: Search is Exa web-only (Search.tsx:10) and the command palette only navigates (command-palette.tsx:85-103). 590 knowledge documents, 19,110 chunks, 246 records, and 1,303 signals are unreachable from inside the app — the operator tabs out to Claude or SQL to answer "what do we know about X".

Change: a search_workspace Postgres function (tsvector over record fields/titles, document titles/summaries, signal titles) exposed as an "Internal" tab in Search and as record results in the command palette, entity-scoped via move 1. Embeddings already exist for deeper retrieval later.

Cost of not doing: the OS never becomes the point of recall; every lookup costs a context switch and agents can't cite a shared index.

First step — this week

Write the search_workspace(query text, entity text default null) SQL function unioned over the three sources with type + rank columns; wire it behind a second tab in Search.tsx. FTS first; skip embeddings until it proves out.

Seed the database with a schema matching the distribution SOPs and backfill the ~6 known in-flight assets (from departments/distribution/assets/) as records; add asset links to the 4 distribution workflows' related_databases.

07 · Explicitly cut

Looks worth doing. Isn't.

✕
Multi-user auth & permission tiers
There is one human operator plus agents. The anon-key RLS scoping already models that boundary. Build auth when a second human exists, not before — it taxes every table and every MCP tool for zero current users.
✕
Replacing the vault with in-app documents
Vault↔Supabase sync already runs at the OS level, and Claude/Fiona/Codex all work the vault natively. Move 4 makes vault-derived knowledge findable in-app; owning the files adds migration risk and breaks three agents' muscle memory.
✕
Generalizing Inbox/Monitors for GoKart
GoKart doesn't run watch-domain intel monitoring — its workflows are client research and content production. Generalizing a surface only one entity uses is effort spent making the wrong thing multi-entity. Entity-scope it (move 1) and stop.
✕
Auto-graph generation & graph expansion
76 nodes of manual graph use says it's an analysis canvas, not a bottleneck. The unused POLE tables are the structured layer worth adopting first; auto-graphing on top of an unadopted evidence model builds the roof before the walls.
✕
A GoKart reports/analytics surface before GoKart has records
Dashboards over empty tables are demo-ware. Seed the databases and one registered workflow first (90-day phase 3); the existing Reports and Home-pin machinery will then serve GoKart with no new build.
✕
Exa Websets / Monitor push delivery, cross-platform builds
Already correctly out of scope. Nothing in the multi-entity thesis changes that call; polling monitors are proven at current volume.

08 · 90-day sequence

Order is dictated by one dependency

Days 1–30

Entity becomes structural

  • Week 1: entities table + FK columns + registry normalization (move 1 first step); regenerate CLAUDE.md to match the real build — every agent session currently starts with a wrong map.
  • Entity filter in the app shell scoping Databases, Agent Work, Workflows, Projects.
  • Read-only MCP tools: list_databases, query_records (move 2 first step).
  • Reciprocity RPC lands (move 3 first step) so writes are safe before write tools exist.
Days 31–60

Agents get hands; GZS gets its pipeline

  • Write-side MCP tools (create_record, update_record, link_records) routing through the RPC layer, emitting work_events.
  • Assets database seeded and live for GZS distribution (move 5); runs + receipts recorded on real briefs and expertise pages.
  • CRM/Introducers get pipeline views + agent-driven updates — first fully agent-operable revenue surface.
Days 61–90

Recall, then the second entity

  • Internal search shipped: search_workspace FTS + Search tab + palette results (move 4).
  • Shared contracts package extracted; MCP and app consume one implementation (move 3 phase 2).
  • GoKart onboarding as the acceptance test: Prospects/Products/Content databases, one registered NeuroDiv content workflow, nav scoped to GoKart showing only GoKart. If this takes more than a week, the entity layer isn't done.
The dependency that dictates the order: every downstream artifact — MCP record tools, the Assets pipeline, internal search, GoKart's databases — writes into or filters by the entity dimension. Retrofit it after seeding GoKart data and you migrate everything twice; build it first and each later move is additive. That's why move 1 is a week-1 migration, and GoKart onboarding is deliberately last: it's the test you run against a finished chassis, not the way you discover the chassis is missing.
